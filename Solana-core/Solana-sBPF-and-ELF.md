# Solana sBPF & ELF — How Solana Programs Work Under the Hood

> Research notes from a deep-dive session into how Solana compiles, packages, and executes on-chain programs.

---

## The Big Picture

When you write a Solana program in Rust, it goes through a pipeline before it can run on-chain:

```
Rust source (.rs)
    → cargo build-sbf
        → sBPF bytecode
            → wrapped in ELF (.so file)
                → deployed to program account (via BPF Loader)
                    → executed by Solana runtime (rbpf VM)
```

---

## What is sBPF?

**sBPF (Solana BPF)** is Solana's customised dialect of eBPF — a 64-bit RISC-like instruction set originally designed for in-kernel packet filtering in Linux.

Solana diverged from vanilla eBPF in several important ways:

- Uses a **different calling convention** — r1–r5 for arguments, r0 for return value, r6–r9 callee-saved, r10 is the read-only frame pointer
- Uses **`syscall` instructions** instead of eBPF kernel helper calls — this is how programs talk to the Solana runtime (e.g. `sol_log_`, `sol_invoke_signed_`)
- **Fixed stack frames** of 4096 bytes per call frame — no dynamic stack growth
- **Compute unit metering** instead of static loop bounds — programs can loop freely but each instruction costs compute units
- Instructions are **64-bit wide** (8 bytes each): opcode | dst/src registers | 16-bit offset | 32-bit immediate

### A single sBPF instruction layout

```
[opcode: 8b] [dst_reg: 4b | src_reg: 4b] [offset: 16b] [imm: 32b]
```

## What is ELF?

**ELF (Executable and Linkable Format)** is the *container format* that wraps the sBPF bytecode. Think of it as a book:

> **ELF is the book** — it has a table of contents (section headers), chapters (.text, .rodata, .dynsym...), and metadata on the cover (ELF header).
>
> **sBPF bytecode is the story written inside** — specifically in the `.text` chapter, in a language only the runtime understands.
>
> **Solana runtime (rbpf) is the reader** — it opens the book, reads the table of contents, finds the story, verifies it, then either reads line by line (interpreter) or memorises and recites at full speed (JIT).

### Key ELF header fields for Solana

| Field | Value |
|---|---|
| `e_machine` | `247` (EM_BPF) — identifies this as BPF |
| `e_entry` | RVA of the `entrypoint` symbol |
| `e_type` | `ET_DYN` (shared object / PIC) |
| Class | `ELFCLASS64` |

### Key ELF sections inside the `.so`

| Section | Contents |
|---|---|
| `.text` | The actual sBPF instructions (8 bytes per instruction) |
| `.rodata` | String literals — your `msg!()` strings live here |
| `.data` / `.bss` | Mutable globals (heavily restricted in Solana) |
| `.rel.dyn` / `.dynsym` | Relocations and dynamic symbol table — resolved at load time |
| `.debug_*` | DWARF debug info — stripped for on-chain deploy, present in dev builds |

## It's One File, Not Two

A common misconception: `cargo build-sbf` produces **one `.so` file**, not two separate files. The sBPF bytecode is not a separate artifact — it lives *inside* the ELF.

```
my_program.so        ← this IS the ELF file
├── .text            ← contains the sBPF bytecode
├── .rodata
├── .dynsym
└── ...
```

sBPF = the instructions. ELF = the container. One file, two concepts.

## What About the IDL?

The **IDL** (Interface Definition Language, a `.json` file) is a completely separate thing — generated by the **Anchor framework**, not by the compiler.

| File | Generated by | Contains | Used for |
|---|---|---|---|
| `my_program.so` | `cargo build-sbf` | ELF + sBPF bytecode | On-chain deployment |
| `my_program.json` | Anchor framework | Instruction/account schema | Client-side SDK generation |

If you're not using Anchor, no IDL is generated — just the `.so`.

## Deployment: Who Does What

```
solana program deploy my_program.so
```

This command does more than copy a file — it sends a series of on-chain transactions.

### The deploy flow

1. **CLI** breaks the `.so` file into chunks
2. Sends transactions to the **BPF Loader** (a native Solana program at a fixed address)
3. Loader writes the raw ELF bytes into the **program account's data**
4. Marks the account as **executable**

The BPF Loader is itself just an on-chain program. This is why deployment costs SOL — you're paying for transactions and for rent-exempt storage of the ELF bytes on-chain.

### The full cast

| Role | Who | Analogy |
|---|---|---|
| `solana program deploy` | Developer | Author submitting the book to a library |
| BPF Loader | Solana's built-in loader program | Librarian receiving and shelving it |
| Client sends transaction | End user | Someone requesting the book |
| rbpf runtime | Solana validator | Reader actually reading and executing it |

## How the Runtime (rbpf) Executes a Program

When a transaction invokes your program, the `solana-rbpf` VM does the following:

1. **Parse ELF** — validate the header, walk section/program headers, extract `.text`
2. **Apply relocations** — resolve symbols like `sol_log_` to their syscall numbers, patch call targets
3. **Verify bytecode** — check for invalid instructions, out-of-bounds memory access, register misuse
4. **JIT compile** (validators) — translate sBPF → native x86-64 for fast execution
5. **Execute** — set up initial register state (r1 = pointer to `SolParameters`, r2 = parameter count), call the entry point, track compute units

### Two execution modes

| Mode | Used by | Description |
|---|---|---|
| JIT | Validators (hot path) | sBPF compiled to native x86-64 — fast |
| Interpreter | Simulation / fallback | sBPF executed instruction by instruction |

---

## Why Programs Can't Have Mutable Globals

Programs are stored as raw ELF bytes in a read-only account. The runtime does not maintain per-execution copies of `.data`. Any mutation to a global would persist across transactions — a consensus nightmare.

This is why **Solana programs are effectively pure functions over accounts**. All state lives in account data passed in as arguments, not in the program binary itself. The ELF is always loaded read-only.

## Key Mental Model Summary

```
sBPF    = the instruction set (what the code says)
ELF     = the file format    (how the code is packaged)
.so     = one file           (ELF container holding sBPF bytecode)
IDL     = Anchor only        (separate JSON, not part of compilation)
Loader  = BPF Loader program (called by developer at deploy time)
Runtime = rbpf VM            (called by validators at execution time)
```

## Quick Reference

```bash
# Build a Solana program
cargo build-sbf

# Output location
target/deploy/my_program.so
target/deploy/my_program-keypair.json

# Deploy on-chain
solana program deploy target/deploy/my_program.so

# Inspect ELF sections (requires llvm-objdump from Solana toolchain)
llvm-objdump -h target/deploy/my_program.so

# Dump sBPF disassembly
llvm-objdump -d target/deploy/my_program.so
```
