# Solana Internals — Accounts, PoH & Tower BFT

> Research notes covering how Solana stores state, orders transactions, and reaches consensus across validators.

---

## The Question That Started This

From the previous session we established that **Solana programs are pure functions** — the ELF (program binary) is read-only and never changes. All state lives in accounts passed in as arguments.

That raised an immediate question: **where are these accounts actually stored?**

---

## Accounts — Where State Lives

### The core rule

> **The book (ELF) never changes. Only the notepads (accounts) do.**

The program is the recipe. The accounts are the ingredients and the dish. You don't rewrite the recipe every time you cook — you just get different ingredients each time and produce a different result. That's what "pure function over accounts" means — same program, different accounts in → different accounts out.

### Why the program binary (ELF) is read-only

Programs are stored as raw ELF bytes in a read-only account. The runtime does not maintain per-execution copies of `.data`. If the book *could* be written in — validator 1 writes "balance = 500" on page 3, validator 2 sees a different value. Across thousands of validators worldwide, they'd all see different versions. That's a **consensus nightmare** — the network can never agree on program state.

So the rule is enforced at the architecture level: the ELF is always loaded read-only.

### Where accounts physically live

Every validator maintains a local copy of the entire account state — this is called **AccountsDB**.

```
Validator machine (physical server)
└── AccountsDB
    ├── account: wallet address ABC     → { balance, owner, data[] }
    ├── account: program address XYZ   → { executable=true, elf bytes }
    ├── account: token account 123     → { balance, mint, owner }
    └── ... (millions of accounts)
```

This is stored on the **validator's disk and RAM** — not offchain in the web2 sense, but literally files on the validator's filesystem.

### Solana's AccountsDB internals

Solana uses a custom storage engine with a specific design:

- Accounts stored in **append-only flat files** called account storage files on disk
- Hot accounts (frequently accessed) cached in **RAM**
- A pubkey **index** maps each address → where on disk that account's latest data is
- Old versions cleaned up by a background **accounts shrink process**

```
/validator-ledger/
└── accounts/
    ├── 0.0     ← append-only storage file (slot 0)
    ├── 1.0     ← slot 1
    ├── 2.0
    └── ...     ← millions of these
```

### How consistency is maintained across validators

Every validator on the network holds an identical copy of AccountsDB. After every block, all validators run the same transactions and arrive at the same new account states — enforced by consensus.

> Every librarian in every branch has an **identical notepad ledger**. When a reader updates their notepad, the head librarian (leader validator) announces the change, and every other librarian updates their own copy to match. If a validator's copy drifts — it gets ignored by the network.

### Summary

| Where | What is stored |
|---|---|
| Validator RAM | Hot accounts, active slots, current epoch state |
| Validator disk (AccountsDB) | Full account state — all pubkeys, balances, data |
| Every validator | A full identical copy of all of the above |

"On-chain" physically means **on every validator's disk**, kept in sync by consensus. There is no single server — the same state replicated across thousands of machines simultaneously.

## PoH — Proof of History

### The problem it solves

In distributed systems, agreeing on the **order and time** of events is hard. In Bitcoin and Ethereum, validators gossip timestamps to each other — but you can't trust a timestamp from a stranger on the internet.

Solana's solution: **don't trust clocks, create a cryptographic clock.**

### How it works

PoH is a continuous **SHA-256 hash chain** where each hash takes the previous output as input:

```
hash_1 = SHA256("genesis")
hash_2 = SHA256(hash_1)
hash_3 = SHA256(hash_2)
hash_4 = SHA256(hash_3)
...
hash_n = SHA256(hash_n-1)
```

SHA-256 is intentionally **sequential** — you cannot compute hash_1000 without computing hashes 1 through 999 first. There is no shortcut.

So if someone shows you hash_1000000, you **cryptographically know** that a certain amount of real time passed to produce it — because you know roughly how long one SHA-256 takes on hardware.

### Transactions get stamped into the chain

```
hash_500 = SHA256(hash_499 + transaction_data)
```

That transaction is now permanently ordered in time relative to everything else — without validators needing to gossip about timestamps.

### The analogy

> PoH is like a **metronome that writes its own proof** — every tick produces a hash that proves the previous tick happened before it. The leader validator holds the metronome, and everyone else can verify the rhythm after the fact.

### Why this makes Solana fast

Validators don't need to agree on time before processing — the order is already cryptographically proven. They can verify the PoH sequence quickly and in parallel, and immediately start voting on blocks.

## Tower BFT — Consensus

### What PoH doesn't solve

PoH handles **ordering and time**. It does not handle **agreement** — making sure all validators commit to the same version of history. That's Tower BFT's job.

### What is BFT?

BFT stands for **Byzantine Fault Tolerance** — a property from distributed systems theory. A network is BFT if it can reach consensus even when some nodes are lying, faulty, or malicious — as long as fewer than ⅓ of nodes are bad actors.

Tower BFT is Solana's customised version of PBFT (Practical Byzantine Fault Tolerance).

### The core mechanism — exponential lockout

Validators **vote** on blocks. The longer you keep voting on a particular fork, the more **locked in** you become:

```
Vote on block A once   → locked out of switching for  2 slots
Vote on A again        → locked out for                4 slots
Vote again             → locked out for                8 slots
Vote again             → locked out for               16 slots
...doubles every time
```

This is called **exponential lockout**.

If you try to switch to a different fork after many votes, you **forfeit your staked SOL** (slashing). The more you have committed to a chain, the more it costs to abandon it.

### The analogy

> Tower BFT is like a **group of judges voting on a verdict**. The longer everyone has been saying "guilty", the harder it becomes to suddenly switch to "not guilty" — because you would have to explain why you changed your mind after so many rounds, and you would lose your credibility (stake) for doing so.

### Finality

Once **⅔ of staked SOL** has voted on a block, it is considered confirmed. Once lockouts reach a certain depth, the block is **finalized** — mathematically impossible to revert without ⅓+ of all stake forfeiting their SOL simultaneously.

## How PoH and Tower BFT Work Together

They solve different halves of the same problem:

| | Solves | How |
|---|---|---|
| **PoH** | Time and ordering | Cryptographic hash chain — proves sequence without trusting clocks |
| **Tower BFT** | Agreement | Exponential lockout voting — makes validators commit and penalises switching |

### The full flow

```
Leader validator
    → runs PoH continuously (produces ordered hash chain)
    → stamps transactions into the chain as they arrive
    → broadcasts blocks to all other validators

Other validators
    → verify the PoH sequence (fast, parallelisable)
    → replay transactions and update their AccountsDB
    → vote on blocks via Tower BFT
    → once ⅔ of stake votes → block confirmed → accounts updated
```

## Connecting Everything Back

```
Program (ELF)          → read-only, stored in a program account
Accounts               → all mutable state, passed in per transaction
AccountsDB             → where accounts physically live (every validator's disk)
PoH                    → proves the order transactions happened in
Tower BFT              → ensures all validators agree on that order
```

The entire architecture flows from one principle: **make validators agree on state without trusting each other**, using cryptography (PoH) and economic incentives (Tower BFT slashing) instead.

## One Line Summary Each

**Accounts** — where all Solana state lives; passed into programs like function arguments, never baked into the program binary.

**AccountsDB** — Solana's custom on-disk storage engine; every validator holds a full identical copy.

**PoH** — a cryptographic clock that proves the order of events without trusting anyone's timestamp.

**Tower BFT** — a voting system where the longer you commit to a version of history, the more it costs to change your mind.
